## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Upstream GitHub repo → local vendored copy | Vendored code becomes our responsibility once copied; no automatic upstream-security-fix flow |
| Cloud Function runtime → GCP billing API | The function holds an IAM role with `billing.resourceAssociations.delete` permission; misuse disables billing |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-00-07 | Tampering | Vendored upstream source | mitigate | Record exact upstream commit SHA in NOTICE and `main.py` banner so any modification is traceable; vendor by copy (D-04) so upstream account compromise after vendoring cannot push code into our repo |
| T-00-08 | Denial of service | Kill-switch fires and disables billing on the live project | accept-with-mitigation | This is the entire point of the kill switch — DoS-by-design as a budget circuit breaker. Recovery procedure documented in `infra/kill-switch/README.md` (`## ⚠ This is a one-way door` section). Budget alerts at 50/90/100% (plan 04) provide warnings before the 100% threshold triggers the function. |
| T-00-09 | Elevation of privilege | Kill-switch function's service account has `roles/billing.projectManager` (required to call `updateBillingInfo`) | mitigate | Plan 04 grants the role to the function's identity only, with least-privilege scope (the project only, not the billing account). The role's only effective action is the one this function takes. |
| T-00-SC | Tampering | Vendored upstream | mitigate | Source URL + commit SHA + Apache-2.0 license recorded; future upstream changes are not auto-pulled. No `git submodule` (D-04). |
